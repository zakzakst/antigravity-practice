# お気に入りボタンの実装

## Story
ユーザーは書籍の詳細ページ上で、その本を「お気に入り登録」または「お気に入り解除」のトグル操作ができる。なぜなら、後でマイページからすぐに見返したり、自分の好みの本をストックしておく管理機能として活用できるから。

## 要件
- 「本の詳細ページ」にお気に入りボタンを配置すること。
- ログイン中の一般ユーザーまたは管理者のみボタンを利用可能な状態にする。未ログインユーザー（ゲスト）には、ボタンを非表示にするか、押下時にログインページへ誘導する。
- ページアクセス時、ユーザーがすでにその本をお気に入り登録しているか状態を取得し、ボタンのデザイン（テキストや色など）を出し分けること。（例：「♡ お気に入りに追加」と「♥ お気に入り解除」などでトグルする）

## UI設計 / モック情報
- `book-app/mock/detail.html` の `btn-secondary` ボタンをベースにコンポーネントとして切り出す。
- 未お気に入り状態と、お気に入り済み状態で見た目の変化（アイコンや背景色、テキスト変更など）が切り替わるよう実装する。

## API連携仕様
- **詳細取得時の初期化**:
  - `GET /api/books/{id}` などのレスポンスを通じて「現在のログインユーザーが該当書籍を登録済みか（`isFavorited: boolean` 等）」の初期状態を受け取る。（※要バックエンド側のフラグ追加対応）
- **お気に入り追加実行時**:
  - `POST /api/books/{id}/favorite` （Bearer Token必須）へリクエストを行う。
- **お気に入り解除実行時**:
  - `DELETE /api/books/{id}/favorite` （Bearer Token必須）へリクエストを行う。

## 作業計画 (Implementation Plan)

### バックエンド (Backend)
1. **DB設計/マイグレーション**: 既存の実装にない場合、`favorites` 中間テーブル（id, book_id, user_id, created_at 等）を作成するテーブル定義とマイグレーションを用意する。
2. **API実装 (POST/DELETE)**:
   - `POST /api/books/{id}/favorite`: JWT等の認証情報から取得した `user_id` とパスの `book_id` を紐付けてDBに保存。重複登録は防止する。
   - `DELETE /api/books/{id}/favorite`: 同様に紐付き情報をDBから削除する。
3. **API連携の拡張 (GETの改修)**:
   - 書籍詳細を返す `GET /api/books/{id}` の処理内で、リクエストヘッダに認証トークンがあれば、そのユーザーが該当本をすでにFavorite済みかDBを調べ、レスポンスのプロパティに `isFavorited: true/false` を追加で返すように修正する。
4. **テスト**: 未ログイン時に401/403エラーが返ること、およびお気に入りのトグルが行えることを検証する。

### フロントエンド (Frontend)
1. **UIとステート(State)の定義**:
   - `isFavorited` (boolean) と `isLoading` (boolean) といった内部状態を持ち、現在のAPI返却値を初期値としてバインディングする。
2. **未ログイン時のハンドリング**:
   - グローバルなユーザー認証状態を参照し、ユーザー情報が無ければボタンをレンダリングしない、もしくはクリック時にアラート/ログイン画面へリダイレクトする処理を入れる。
3. **アクションのクリック制御**:
   - ボタンクリック時、`isFavorited` ならば `DELETE` を呼び出し、そうでないならば `POST` を呼び出す。
   - リクエスト中は `isLoading: true` にして連続クリック（連打）を防ぐ。
   - 成功レスポンス後、`isFavorited` を反転させる（あるいはUIを再フェッチする）。
