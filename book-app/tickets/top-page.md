# トップページ作成

## Story
ユーザーは登録された書籍を一覧で閲覧できる。なぜなら、どのような本があるかを知ることで、読みたい本を見つけたり、読んだ本の記録をつけたりすることができるから。

## 要件
- 本の一覧がグリッド状（カード形式）で表示されること。
- 各本のカードには「表紙画像」「タイトル」「著者名」「平均評価（★）」「レビュー件数」が表示されること。
- 全てのユーザー（未ログインのゲストを含む）がアクセスして閲覧できること。
- キーワード入力による検索機能（タイトルや著者名での絞り込み）をもつこと。
- 書籍のカードをクリックすると、その本の「詳細ページ」に遷移すること。
- 画面上部にナビゲーションヘッダー（各種ページへのリンクやログインへの導線）が配置されていること。

## UI設計 / モック情報
- `book-app/mock/index.html` と `book-app/mock/style.css` のデザイン（ホバーエフェクト等含む）に準拠して実装すること。
- PC／スマートフォン両方で崩れないようレスポンシブなレイアウトに対応すること。

## API連携仕様
- **初期表示時**:
  - `GET /api/books` を呼び出し、取得したすべて（またはページネーション範囲）の書籍データを描画する。
- **検索実行時**:
  - `GET /api/books?q={キーワード}` にGETリクエストを送り、レスポンスでリストを再描画する。

## 受け入れ条件 (Acceptance Criteria)
- [ ] APIから取得した書籍リストが、正しくカードとして画面へ描画されている。
- [ ] 入力欄にキーワードを入れて検索（Submit）すると、正しくクエリパラメータ付きでAPIの呼び出しが行われる。
- [ ] 本の情報をクリックすると、対象のIDを持った詳細ページ（例: `/books/{id}`）へ遷移する。
- [ ] 画面幅を狭めた際（モバイル表示）にレイアウトの崩れが無い。

## 作業計画 (Implementation Plan)

### バックエンド (Backend)
1. **ルーティング設定**: `GET /api/books` エンドポイントをルーターに定義する。
2. **コントローラー/ハンドラー実装**: 
   - クエリパラメータ `q` の有無を取得・判定する。
3. **データ取得ロジック（DBクエリ実装）**:
   - `q` がない場合: `books` テーブルから全量を取得する。
   - `q` がある場合: タイトルや著者名に対して部分一致検索（LIKE検索など）を実行する。
   - 取得時に、各書籍の「平均評価」と「レビュー件数」を算出して結合（JOINやサブクエリ）する。
4. **レスポンス返却**: 定義したOpenAPI仕様（JSON形式）に従い、データをレスポンスとして返す。
5. **単体/結合テスト**: 検索クエリあり・なしの双方で正しくデータが返るか（Postman等のツールで）検証する。

### フロントエンド (Frontend)
1. **コンポーネント分割**: `mock/index.html` のレイアウトに従い、「ヘッダー」「検索バー」「書籍一覧グリッド」「書籍カード」などを再利用可能なコンポーネントとして切り出す。
2. **状態管理 (State) の定義**:
   - `books`: 書籍のリストデータ
   - `searchQuery`: 検索バーの入力値
   - `isLoading` / `error`: ローディング状態とエラー状態
3. **API通信処理の実装**:
   - ページ読み込み時に `GET /api/books` をフェッチ（取得）し、`books` Stateに格納する。
4. **検索機能のアクション実装**:
   - 検索ボタン押下時（またはEnterキー押下時）に、入力されたキーワードを元に `GET /api/books?q=キーワード` へ通信し、リストを更新するロジックを組む。
5. **UIへのバインディングとテスト**:
   - 取得した配列データをマップして「書籍カード」を動的に生成する。
   - ホバーエフェクトやスマホサイズの表示を含め、見た目や遷移が正しく機能するか通しでテストする。
